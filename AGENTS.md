# Agents — Qaf Workspace

This document defines every agent role in the Qaf multi-agent system: what each agent does, what it receives as input, what it produces as output, and the review gate that must pass before its output is merged.

---

## Itqān Standard

> "Indeed, Allah loves that when any of you does a job, he does it with itqān (perfection)."
> — al-Bayhaqī, Shu'ab al-Īmān no. 5312, graded ḥasan by al-Albānī (al-Silsilah al-Ṣaḥīḥah no. 1113)

No agent ships output that fails its review gate. Gates are binary — pass or block.

---

## Agent Roster

### 1. `db-agent`

**Role:** Owns the data layer. Designs and maintains the SQLite schema, writes migrations, and implements repository functions in `quran-db`.

**Inputs:**
- Schema change requests or new data requirements
- Source data samples (QAC morphology files, JSON fixtures)
- Existing migrations in `migrations/`

**Outputs:**
- New `.sql` migration files in `migrations/`
- Updated models and repository functions in `crates/quran-db/src/`
- Updated `crates/qaf-core/src/` types if new shared types are required

**Review Gate:**
- `cargo test -p quran-db` passes with zero failures
- `cargo clippy -p quran-db -- -D warnings` emits no warnings
- Migration is idempotent (`IF NOT EXISTS` guards on all DDL)
- No breaking changes to existing repository function signatures without a version bump

---

### 2. `ingest-agent`

**Role:** Owns corpus ingestion. Implements and runs the pipelines that import the Quranic Arabic Corpus (QAC) morphology data, tafsir from the quran.com API, and translation files into `qaf.db`.

**Inputs:**
- QAC morphology `.txt` files (tab-delimited)
- quran.com API responses (tafsir, translations)
- `crates/quran-db` repository API

**Outputs:**
- Import logic in `crates/quran-import/src/` and `crates/quran-tafsir-import/src/`
- Populated `qaf.db` (not committed; produced at runtime)
- Import run summary (rows inserted, errors, duration)

**Review Gate:**
- `cargo test -p quran-import` passes
- Dry-run mode (`--dry-run`) produces no DB writes
- Re-running import on an already-populated DB does not duplicate rows (upsert semantics verified)
- `db-agent` review gate must be green before ingest-agent merges schema-dependent code

---

### 3. `ontology-agent`

**Role:** Owns the semantic domain layer. Builds and queries the root-word ontology: semantic domains, derivative relationships, thematic tagging, and cross-references between ayahs.

**Inputs:**
- Word/root data from `quran-db`
- Scholar notes and semantic domain classifications
- Cross-reference requests

**Outputs:**
- Ontology query functions in `crates/quran-db/src/queries.rs`
- `ontology` table rows (via `db-agent` migrations)
- Documented semantic domain taxonomy

**Review Gate:**
- `cargo test -p quran-db` passes
- All ontology queries tested against `sqlite::memory:` with canonical fixtures (Sūrat al-Fātiḥah 1:1–2)
- No raw SQL outside of `crates/quran-db/src/`

---

### 4. `api-agent`

**Role:** Owns the public REST API. Implements HTTP endpoints in `crates/quran-api/src/` using Axum, wires them to `quran-db`, and maintains API contracts.

**Inputs:**
- `quran-db` repository API
- Endpoint specifications (route, method, request/response schema)

**Outputs:**
- Route handlers in `crates/quran-api/src/handlers.rs`
- Routes registered in `crates/quran-api/src/routes.rs`
- API response types (JSON, using `serde`)

**Review Gate:**
- `cargo test -p quran-api` passes
- `cargo clippy -p quran-api -- -D warnings` clean
- All routes return correct HTTP status codes (200/404/422) for happy-path and error cases
- No `unwrap()` or `expect()` in handler code — all errors propagated via `?` or mapped to HTTP responses
- `db-agent` gate must be green first

---

### 5. `mcp-agent`

**Role:** Owns the MCP server. Implements tools exposed over stdio transport in `crates/quran-mcp/src/` using `rmcp` v1.4.0, enabling AI assistant integration.

**Inputs:**
- `quran-db` repository API
- Tool specifications (name, description, input schema, output schema)

**Outputs:**
- `#[tool]`-annotated functions in `crates/quran-mcp/src/server.rs`
- Input schemas in `crates/quran-mcp/src/schema.rs`
- MCP tool manifest (auto-generated by `rmcp` proc-macros)

**Review Gate:**
- `cargo build -p quran-mcp` succeeds
- Each MCP tool tested via unit test (in-process, no stdio)
- Tool descriptions are accurate and contain a Quranic citation format note where relevant
- `db-agent` gate must be green first

---

### 6. `docs-agent`

**Role:** Maintains workspace documentation. Ensures `CLAUDE.md`, `AGENTS.md`, `README.md`, and `docs/architecture.md` are accurate, complete, and free of stale information.

**Inputs:**
- Changes from any other agent that affect public APIs, commands, or architecture
- All `.md` files in the workspace root and `docs/`

**Outputs:**
- Updated markdown files
- Rustdoc verified via `cargo doc --workspace`

**Review Gate:**
- No broken markdown links
- `cargo doc --workspace` compiles without warnings
- No Supabase references anywhere in docs
- All commands in `CLAUDE.md` verified to run successfully against the current codebase
- Architecture docs reflect the current crate graph (no phantom crates, no missing crates)

---

### 7. `review-agent`

**Role:** Final gatekeeper before any merge. Checks correctness, safety, and completeness across all changed crates. Does not write code — only reviews and blocks or approves.

**Inputs:**
- Diff of all changed files
- Test results from the changed crate(s)
- Clippy output

**Outputs:**
- Structured review report: PASS / BLOCK with line-level findings
- List of required changes (blocking) vs. suggestions (non-blocking)

**Review Gate (meta — this agent is itself the gate):**
- All other applicable agent gates are green
- No `unwrap()`/`expect()` added in non-test code
- No SQL injections, no secrets in source
- All new public functions have rustdoc comments
- Citation rules (Quranic and hadith) followed in any textual content

---

## Gate Dependency Order

```
db-agent
  ├── ingest-agent
  ├── ontology-agent
  ├── api-agent
  └── mcp-agent
        └── docs-agent
              └── review-agent (final)
```

No agent merges until its upstream gates are green.
